I'm building a travel day-planner that uses a weighted A* search algorithm to schedule a full day of activities. I need you to help me generate realistic, accurate JSON test data for specific cities.
The algorithm works like this: each node represents departing an activity at a specific time. The scheduler picks activities to maximize a "quality of time" metric — time spent at a 5-star activity is 0% dead time, a 3-star activity is 40% dead time, and travel/waiting is 100% dead time. It also tracks a "fullness" meter that decays over time and gets replenished by eating, so meal opportunities need to be well-distributed.
Here's what matters for each field and why:
id: Sequential integer starting at 1.
name: Commonly used name.
type: One of: museum, park, landmark, restaurant, event, boat_tour, observation, market, neighborhood, amusement, hike, tour, nightlife, shopping, wellness, beach, sports. The scheduler uses this to ensure variety.
must_visit: Boolean. Only true if I specifically say so. Default false.
rating: 1.0-5.0 with one decimal. THIS IS CRITICAL. Use a realistic spread — tourist traps at 3.0-3.5, solid spots at 3.8-4.2, world-class at 4.5+. Not everything should be 4.5+. The algorithm uses this directly as a quality score.
start_location / end_location: Real street addresses. The algorithm computes a travel time matrix between all addresses. end_location is null unless you physically end up somewhere different (walks, ferries, hikes). Wrong addresses break routing.
arrive_after / leave_before: Per-weekday opening/closing times as HH:MM strings. Null for a specific day means closed. Null for the whole object means no constraint (outdoor landmarks). MUST be accurate — the algorithm eliminates activities you can't reach before close.
optimal_duration: Minutes for an ideal visit. A major museum might be 180, a viewpoint might be 15, a restaurant 60-90. This directly controls how long the scheduler allocates.
minimum_duration: Shortest visit worth making. REQUIRED for anything with event_start_times. A 3-hour hike has minimum 180. A big museum might have minimum 45. A quick landmark might be 10.
event_start_times: Per-weekday arrays of HH:MM start times. ONLY for fixed-schedule activities (shows, tours, ferries, cruises). Null for flexible venues. When set, the algorithm computes departure as start_time + minimum_duration.
food: Null if no food. Otherwise has breakfast/lunch/dinner/snacks sub-objects, each with per-weekday start/end times. The scheduler uses this to replenish a fullness meter. If a place serves food all day, use breakfast/lunch/dinner with appropriate windows, not snacks. Important: if a venue has a notable cafe or restaurant, consider making it a SEPARATE entry with its own rating and food times so the algorithm can independently decide whether to eat there.
estimated_cost_usd: Per person in USD. 0 for free activities. The algorithm enforces a budget constraint.
exclusions: List of IDs. Fill this LAST. Means "if you've done this, skip those." For competing experiences — two harbor tours, three observation decks, similar museums. Must be mutual (if A excludes B, B excludes A).
description: Brief, few sentences.
considerations: Array of practical tips like "book in advance", "cash only", "wear walking shoes".
Some locations or activities may require multiple entries. A museum and it's cafe should be separate entries. A solo museum visit and a museum tour should be separate (and mutually exclusive) entries.
A walk down a street should be separate from the walk up the street, and a ferry ride one way should be separate from the opposite direction (use your own judgement on mutual exclusion).
